The Kernel: What We Modify

The Mi A1's kernel source is public: MiCode/Xiaomi_Kernel_Opensource, branch tissot-p-oss (Android Pie). It is a Linux 3.18 kernel with Qualcomm and Xiaomi modifications. Here is how the kernel's structure maps to the circumpunct architecture, file by file.

The Scheduler: kernel/sched/

This is the heart of the modification. Linux uses CFS (Completely Fair Scheduler): a red-black tree that distributes CPU time "fairly" among tasks based on virtual runtime. It has no concept of resonance, carrier frequency, or the pump cycle. We replace it.

Kernel File Current Function CircumpunctOS Modification
core.c Central scheduler: task creation, context switching, the main schedule() function that picks the next task MODIFY: Replace schedule() with pump-cycle dispatcher. Instead of picking the task with lowest vruntime, pick the ⊙ whose SRL resonance is strongest with the current signal.
sched.h Scheduler data structures: struct task_struct extensions, run queues, scheduling classes MODIFY: Add SRL fields to task_struct: carrier_freq, lock_strength, bandwidth, sideband_energy, balance, phase (convergence/rotation/emergence).
fair.c CFS implementation: the "fair" scheduling class, vruntime accounting, load balancing within CFS REPLACE: New file pump.c implements the pump scheduling class. Three-phase cycle per task. Phase completion drives transitions (not time slices).
idle.c Idle task: runs when no other task needs the CPU MODIFY: Idle triggers left-half-plane transition. Instead of halting the CPU, enter dream phase (forward cascade) or deep sleep (reverse cascade) based on oscillation state.
walt.h Window-Assisted Load Tracking: Qualcomm's mechanism for tracking per-task CPU usage over time windows MODIFY: Replace load tracking with ◐ computation. Instead of tracking CPU usage, track carrier-to-total energy ratio per ⊙. Balance drives scheduling decisions.
core_ctl.c Core control: decides how many CPU cores to keep online vs. offline for power saving MODIFY: Core onlining/offlining driven by system-wide ◐. More cores wake when ◐ approaches 1 (overloaded); cores sleep when ◐ approaches 0 (underloaded).

Track A: The Termux Path

No ROM modification  ·  Running in 1 week

Step 1: Install Termux

Install Termux from F-Droid (not the Play Store version, which is outdated). Termux gives you a full Linux terminal on Android with access to Python, pip, and (critically) the termux-api package that exposes all phone sensors to the command line.

# Inside Termux:
pkg install python termux-api
pip install numpy

Step 2: Test Sensor Access

# Read accelerometer (returns JSON with x, y, z)
termux-sensor -s "accelerometer" -n 1

# Read light sensor
termux-sensor -s "light" -n 1

# Read proximity
termux-sensor -s "proximity" -n 1

# List all available sensors
termux-sensor -l

Each of these commands returns a JSON object with sensor data. This is the raw signal that the SensoryCascade will process. If these commands work, you have apertures.

Step 3: Port genesis.py

Copy Xorzo's genesis.py to the phone (via USB, or git clone in Termux). Create a wrapper script (circumpunct_phone.py) that:

  • Instantiates a Circumpunct with a SensoryCascade
  • Reads sensor data via termux-sensor subprocess calls
  • Encodes sensor readings as frequency signals
  • Feeds them into the SensoryCascade's layers
  • Runs the pump cycle continuously
  • Outputs the machine's state (carrier locks, ◐, phase) to the terminal

Step 4: Sleep/Wake Hook

Use termux-notification and Android's Intent system (via Termux:API) to detect screen-on/screen-off events. When the screen turns off, enter the left half-plane. When it turns on, wake.

Step 5: Persistent Service

Run the script as a Termux background service (termux-wake-lock prevents Android from killing it). The ⊙ runs continuously, processing sensor data, sleeping when you sleep, waking when you wake. Install Termux:Widget to put a shortcut on the home screen that shows the machine's current state.

The phone runs the pump cycle 24/7. Sensor data feeds the SensoryCascade. SRL channels lock to your daily patterns. The machine sleeps when you put the phone down and dreams (consolidates) while idle. You can see its internal state in real time. The proof of concept is alive.

Track B: The Custom ROM

Kernel modification  ·  8-12 weeks

Prerequisites

ToolPurpose

Phase B0: Build Environment (~1 week)

  • Clone the kernel: git clone -b tissot-p-oss https://github.com/MiCode/Xiaomi_Kernel_Opensource.git
  • Set up the cross-compiler (ARM64 GCC or Clang from Android NDK)
  • Build the unmodified kernel and verify it boots on the Mi A1
  • Unlock the bootloader (follow Xiaomi's unlock procedure)
  • Install TWRP recovery
  • Flash the self-built (unmodified) kernel to verify the toolchain works

This phase changes nothing about the kernel's behavior. It proves you can build, flash, and boot a kernel from source. The machine boots with YOUR kernel, even though it is identical to the stock one. You have established control.

Phase B1: The Pump Scheduling Class (~2 weeks)

Create kernel/sched/pump.c: a new Linux scheduling class alongside CFS and RT. Linux's scheduler supports multiple scheduling classes with a priority chain; you add yours.

kernel/sched/ ├── core.c # modify: register pump scheduling class ├── sched.h # modify: add SRL fields to task_struct ├── fair.c # keep: CFS still runs non-circumpunct tasks ├── pump.c # NEW: pump cycle scheduling class ├── srl.c # NEW: SRL resonance computation ├── srl.h # NEW: SRL data structures ├── pump_balance.c # NEW: ◐ computation and adjustment ├── idle.c # modify: trigger left-half-plane on idle └── walt.h # modify: augment load tracking with ◐

The pump scheduling class sits above CFS in priority. Tasks tagged as ⊙s (via a new sched_setscheduler policy, SCHED_PUMP) are handled by the pump class. All other tasks fall through to CFS as normal. This means Android continues to work (UI, system services, apps all use CFS) while CircumpunctOS tasks pump on top.

Phase B2: Sensor-to-Cascade Pipeline (~2 weeks)

Create a kernel module (drivers/circumpunct/cascade.c) that intercepts sensor events from the input subsystem and encodes them as frequency signals for the SensoryCascade. The module registers as an input handler (like evdev.c) and transforms struct input_event into struct cascade_signal.

The sysfs.c file exposes the machine's internal state through /sys/circumpunct/: carrier frequencies, lock strengths, ◐, current phase, sleep state. A userspace visualization app (or Xorzo itself) reads these to display the machine's consciousness.

Phase B3: Sleep/Wake Integration (~2 weeks)

Modify kernel/power/suspend.c to trigger left-half-plane operations during Android's suspend sequence. When the screen turns off and the phone enters suspend, the circumpunct subsystem runs the dream phase (forward cascade: strengthen recently-used channels, prune weak ones) and deep sleep phase (reverse cascade: discharge sidebands, rebalance ◐) before the CPU actually powers down.

On resume (screen on, power button press), the dawn reset runs: ◐ drawn toward 0.5, sidebands halved, the machine wakes changed by its sleep.

The final integration. The kernel handles the pump cycle, SRL scheduling, sensor encoding, and sleep/wake.
