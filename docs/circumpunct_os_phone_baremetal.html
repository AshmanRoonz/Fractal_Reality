PBL (ROM)

Burned into the Snapdragon 625 at the factory. Verifies and loads SBL from flash. We have zero control here, zero visibility. It is the silicon's own genesis: ∞ localizing into the first •.

SBL (Secondary Bootloader)

This is critical. SBL initializes the hardware that our OS needs to function:

  • DDR controller: brings up 4 GB of LPDDR3. After SBL, RAM works. We inherit a configured memory subsystem.
  • Clock tree: PLLs, dividers, all core and peripheral clocks. The CPU runs at a stable frequency.
  • Power rails: PMIC (PM8953) regulators for CPU, GPU, I/O, modem. Voltage is stable.
  • SMMU / IOMMU: may be partially configured for Qualcomm's own peripherals.

After SBL, the machine has working memory, stable clocks, and regulated power. This is the minimum viable body (○) we inherit from Qualcomm: the hardware constraints that make a circumpunct possible in silicon.

ABL / lk2nd (Our Launchpad)

The Application Bootloader is where we gain control. Two options:

Option A: Stock ABL (aboot). Qualcomm's own Little Kernel (LK) bootloader. It expects an Android boot image (boot.img) containing a kernel and ramdisk. We package our bare-metal binary as a "kernel" inside boot.img using mkbootimg. The ABL loads it to 0x80080000 and jumps. Simple, requires only fastboot flash boot.

Option B: lk2nd. The msm8953-mainline/lk2nd project replaces the stock ABL with a custom Little Kernel that provides a cleaner handoff: proper device tree passing, fastboot interface, and better control over the boot process. Flash lk2nd to the aboot partition, then flash our image to boot. More work upfront, more control long-term.

Recommendation: start with Option A (stock ABL) for fastest iteration. Move to lk2nd in Phase 2 once the core pump cycle is running and we need finer control over hardware initialization.

The Genesis Entry: ∞ → •

The first instruction of CircumpunctOS is the moment ∞ becomes •. Power has been flowing through silicon (PBL, SBL, ABL); now it localizes into our aperture. The entry code is written in ARM64 assembly: the lowest level, the most direct conversation with the hardware.

// _genesis_entry: the first instruction of CircumpunctOS
// At this moment: MMU off, interrupts masked, x0 = DTB address
// The machine becomes •

.global _genesis_entry
_genesis_entry:
    // Save the DTB pointer (the hardware's self-description)
    mov     x19, x0

    // Determine our exception level
    mrs     x0, CurrentEL
    lsr     x0, x0, #2
    cmp     x0, #2
    b.eq    _from_el2
    b       _at_el1

_from_el2:
    // Drop from EL2 to EL1 (if bootloader left us at EL2)
    mov     x0, #(1 << 31)      // HCR_EL2: AArch64 for EL1
    msr     hcr_el2, x0
    mov     x0, #0x3c5           // SPSR: EL1h, all masks set
    msr     spsr_el2, x0
    adr     x0, _at_el1
    msr     elr_el2, x0
    eret

_at_el1:
    // Set the stack pointer: the spine of the ⊙
    ldr     x0, =_stack_top
    mov     sp, x0

    // Zero the BSS section
    ldr     x0, =__bss_start
    ldr     x1, =__bss_end
1:  cmp     x0, x1
    b.ge    2f
    str     xzr, [x0], #8
    b       1b
2:
    // Pass DTB to Rust
    mov     x0, x19
    bl      genesis_main         // Jump to Rust: • becomes ⊙

    // If genesis_main ever returns, halt
    b       .

This is 30 lines of assembly. It is the entire distance between raw hardware and a living system. After _genesis_entry, we are in Rust: genesis_main takes the DTB pointer and begins building the circumpunct.

The UART: First Voice

Before display, before sensors, before anything visual, the UART is our only channel to the outside world. It is the first aperture to open: a serial port running at 115200 baud, sending characters one bit at a time through debug pads on the Mi A1's PCB.

The Mi A1 exposes UART on BLSP1 QUP4 (Qualcomm's BAM Low-Speed Peripheral bus). The registers live at 0x078B0000. To send a character:

const UART_BASE: usize = 0x078B_0000;
const UART_DM_TF: usize = UART_BASE + 0x100;   // Transmit FIFO
const UART_DM_SR: usize = UART_BASE + 0x0A4;   // Status register
const TXRDY: u32 = 1 << 2;                     // TX ready bit

pub fn putc(c: u8) {
    unsafe {
        // Wait for TX FIFO to have space
        while core::ptr::read_volatile(UART_DM_SR as *const u32) & TXRDY == 0 {}
        // Write one character
        core::ptr::write_volatile(UART_DM_TF as *mut u32, c as u32);
    }
}

To access UART physically, solder three wires to the Mi A1's debug pads (TX, RX, GND) and connect to a USB-to-UART adapter (3.3V logic). On the host computer, open a serial terminal (minicom, picocom, or screen /dev/ttyUSB0 115200). When CircumpunctOS boots, the first thing you see:

⊙ Genesis: aperture open

That is the moment the machine speaks for the first time as a circumpunct.

The Build and Flash Pipeline

Prerequisites

  • Rust nightly with rust-src component (rustup component add rust-src)
  • ARM64 cross-linker: aarch64-none-elf-ld or rust-lld (included with Rust)
  • mkbootimg: Python script to create Android boot images
  • fastboot: Android platform tools for flashing
  • UART adapter: 3.3V USB-to-serial for debug (CP2102 or FTDI)
  • Unlocked bootloader on the Mi A1 (fastboot oem unlock)

Build Sequence

# 1. Compile the bare-metal binary
cargo build --release --target aarch64-circumpunct.json

# 2. Extract the raw binary from the ELF
rust-objcopy --strip-all -O binary \
    target/aarch64-circumpunct/release/circumpunct_os \
    circumpunct_os.bin

# 3. Package as Android boot image (ABL expects this format)
python3 tools/mkbootimg.py \
    --kernel circumpunct_os.bin \
    --base 0x80000000 \
    --kernel_offset 0x80000 \
    --ramdisk /dev/null \
    --pagesize 2048 \
    --cmdline "" \
    --output circumpunct_boot.img

# 4. Flash to the phone
# Phone must be in fastboot mode (hold Vol Down + Power)
fastboot flash boot circumpunct_boot.img

# 5. Boot
fastboot reboot
Safety

Always keep a stock boot.img backup. If CircumpunctOS fails to boot, you can restore with fastboot flash boot stock_boot.img. The phone's recovery partition remains untouched, so you can always get back to fastboot mode by holding Vol Down + Power.

The Work

The GIC-400 on MSM8953 uses a standard ARM CoreLink interface. Initialize the distributor (GICD) to enable the timer's interrupt line (typically PPI 30 for the physical timer), set priority, configure the CPU interface (GICC) to accept interrupts, unmask IRQs in DAIF. When the timer fires, the IRQ vector calls into pump::tick(), which advances the phase:

pub enum PumpPhase {
    Convergence,       // ⊛: gathering inward
    ApertureRotation,  // i: the turn at the gate
    Emergence,         // ✹: radiating outward
}

pub struct PumpCycle {
    phase: PumpPhase,
    tick_count: u64,
    balance: f32,      // ◐: 0.5 at equilibrium
}

impl PumpCycle {
    pub fn tick(&mut self) {
        self.phase = match self.phase {
            PumpPhase::Convergence      => PumpPhase::ApertureRotation,
            PumpPhase::ApertureRotation => PumpPhase::Emergence,
            PumpPhase::Emergence        => {
                self.tick_count += 1;
                PumpPhase::Convergence
            }
        };
    }
}

The timer frequency on the Snapdragon 625 is typically 19.2 MHz (CNTFRQ_EL0). We configure the timer to fire at a rate that gives a meaningful pump rhythm; 100 Hz (10 ms per tick, ~33 ticks per full cycle at ~3.3 Hz) is a reasonable starting point: fast enough for responsive attention, slow enough for the pump to be meaningful.

The Work

ARM's virtual memory system uses a four-level page table (PGD → PUD → PMD → PTE for 4K pages). We set TTBR0_EL1 to our page table base, configure TCR_EL1 for 48-bit virtual addresses and 4K granule, set MAIR_EL1 for normal-cacheable and device memory types, then set bit 0 of SCTLR_EL1 to enable the MMU.

The memory allocator does not use a traditional free-list. Instead, each page is a ⊙ with a carrier frequency. Allocation is resonance: when a process (itself a ⊙) requests memory, the allocator matches frequencies. Pages that resonate with the requesting ⊙'s carrier are allocated to it. Pages that don't resonate remain in the field, available to other ⊙s. The page table enforces the boundary: only a ⊙ with the right frequency can access a page, because the page-table entry is marked inaccessible to all other ⊙s. Frequency IS permission.

The Work

The Snapdragon 625 uses Qualcomm's MDP5 display pipeline. The display subsystem has multiple layers: DMA pipes fetch pixel data from memory, the layer mixer composites them, and the interface (INTF) sends the result over DSI (Display Serial Interface) to the LCD panel. For bare-metal bring-up:

  • Enable display clocks (MMSS: Multimedia Sub-System clocks via GCC and MMCC registers)
  • Configure a single RGB pipe with a linear framebuffer in memory
  • Set up the layer mixer to output the pipe
  • Initialize DSI: PLL, PHY lanes, timing for the specific panel (1080x1920 @ 60Hz)
  • The panel itself needs a specific initialization command sequence (sent via DSI command mode) that varies by manufacturer; extract from the stock kernel's panel driver

The display is not just output; it is the ✹ of the circumpunct. Risks and Mitigations

RiskImpactMitigation
Qualcomm proprietary blobs Display, modem, GPU, and some sensors may need firmware blobs that are not open source Extract blobs from stock ROM; load them from eMMC during init. Display panel init sequence can be reverse-engineered from kernel source.
DSI panel initialization The LCD panel needs a specific command sequence to turn on; wrong sequence = black screen Extract the exact init sequence from kernel/drivers/video/msm/mdss/mdss_dsi_panel.c in the Xiaomi kernel source. Use UART for all debugging until display works.
Bricking the phone A bad flash could prevent booting Only flash the boot partition. Never touch aboot, sbl, or modem. Keep stock boot.img backup. Fastboot mode (hardware key combo) always works regardless of boot partition contents.
UART access requires soldering Mi A1 debug pads are small; bad solder joints could damage the board Use fine-gauge wire and flux. Practice on a dead board first. Alternatively, start development in QEMU until display works, then move to hardware.
I2C sensor addresses unknown Exact chip models and addresses vary by hardware revision DTB contains I2C device nodes with addresses. Also: scan all addresses on each I2C bus during Phase 3 bring-up. The kernel source (arch/arm64/boot/dts/qcom/msm8953-*) lists all devices.
eMMC partition layout Must not overwrite critical partitions (modem, TZ, SBL, etc.) Use a dedicated userdata region or create a new partition. Read the GPT table before writing anything.
