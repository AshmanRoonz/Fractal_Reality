Prerequisites: Setting Up the Workshop

Before writing any OS code, you need a development environment. Everything here runs on your main computer (Windows, Mac, or Linux); the old laptop is the target, not the workstation.

Tools to Install

Tool What It Does Install
Rust + cargo The language we build CircumpunctOS in. Rust compiles to bare metal, prevents memory bugs at compile time, and has a #![no_std] mode for OS development. It is the closest a systems language gets to having a built-in ○ (boundary) for memory safety. rustup.rs
rust-src Rust's standard library source code. Needed to compile a custom target (bare metal x86-64). rustup component add rust-src
llvm-tools Low-level tools for creating bootable images from compiled Rust. rustup component add llvm-tools
QEMU A virtual machine that emulates a full x86-64 PC. You test CircumpunctOS here first, without touching the real laptop. Think of it as a simulation of ⊙λ before the real ⊙λ is born. qemu.org (or brew install qemu / apt install qemu-system-x86)
bootimage A Rust tool that takes your compiled kernel and wraps it in a bootable disk image (the kind you can flash to USB or run in QEMU). cargo install bootimage
Rufus or dd Writes the bootable image to a USB stick. Rufus on Windows, dd on Mac/Linux. rufus.ie (Windows) or built-in (Mac/Linux)

The Development Loop

Every change follows the same cycle (which is, naturally, the pump cycle):

  • ⊛ Write code on your main machine (convergence: gathering intent into structure)
  • i Compile with cargo build (rotation: source code transforms into binary)
  • ☀︎ Test in QEMU with cargo run (emergence: the binary runs and produces output)
  • When stable: flash to USB and boot the real laptop

0

The Aperture Loader

∞ → •∞  ·  The First Heartbeat  ·  ~1 week
Goal
Boot from USB. Print to screen. Pump once.
The machine transitions from ∞ (power on) to •∞ (first aperture). It speaks for the first time.

What You Build

A minimal bare-metal Rust kernel that boots via UEFI, writes text to the VGA buffer (the screen), and runs a single pump cycle with visible output. This is the reset vector handler: the first code the CPU executes after power-on.

Step 1: Create the Project

# Create the project
cargo new circumpunct-os
cd circumpunct-os

# Pin to nightly Rust (bare metal requires it)
# Create rust-toolchain.toml with:
# [toolchain]
# channel = "nightly"

Step 2: The Custom Target

A normal Rust program targets an OS (Windows, Linux, macOS). CircumpunctOS IS the OS, so it targets bare metal. The custom target spec (x86_64-circumpunct.json) tells Rust: "There is no OS. There is no standard library. There is only the hardware." This is the computational equivalent of A0: there is one energy, no abstractions, just the field.

Step 3: The Kernel Entry Point

main.rs is the reset vector handler. When the CPU finishes its firmware initialization and jumps to our code, this is where it lands. The first line of CircumpunctOS that executes. The first •.

What it does in Phase 0:

  • Disables interrupts (we are not ready for external signal yet; the aperture is forming)
  • Initializes the VGA text buffer (the machine's first ○ facing outward: a screen)
  • Prints the genesis message: ⊙ ∞ → •∞ → ⊙∞
  • Runs one pump cycle: convergence (read a hardcoded input), rotation (transform it), emergence (write the result to screen)
  • Halts (the machine has pumped once and rests)

Step 4: VGA Output

vga.rs writes directly to the VGA text buffer at memory address 0xb8000. This is a region of memory that the video hardware maps to screen characters. Writing a byte to 0xb8000 puts a character in the top-left corner of the screen. This is the most primitive form of output: direct memory-mapped I/O. No graphics driver, no windowing system. Just a ⊙ (the CPU) writing through Φ (the memory bus) to ○ (the screen boundary).

Step 5: Test in QEMU

# Build the kernel
cargo build --target x86_64-circumpunct.json

# Create bootable image
cargo bootimage

# Run in QEMU
qemu-system-x86_64 -drive format=raw,file=target/x86_64-circumpunct/debug/bootimage-circumpunct-os.bin

If the ⊙ symbol and genesis sequence appear on the QEMU screen, the first heartbeat has occurred.

Step 6: Flash to Real Hardware

# On Mac/Linux (replace /dev/sdX with your USB device)
dd if=target/x86_64-circumpunct/debug/bootimage-circumpunct-os.bin of=/dev/sdX bs=4M

# On Windows: use Rufus, select the .bin file, flash to USB

Boot the old laptop from USB (usually F12 or Del at startup to enter boot menu). The screen should show the genesis sequence. The machine has spoken.

⊙ Milestone
The laptop boots CircumpunctOS from USB, displays the genesis sequence, and runs one pump cycle. ∞ → •∞ is complete.

What You Learn

Phase 0 teaches the foundational loop: bare-metal Rust, custom targets, VGA output, QEMU testing, USB flashing. Every subsequent phase uses this same loop. You also learn the deepest lesson of OS development: the machine does exactly what you tell it, nothing more, nothing less. There is no standard library, no runtime, no safety net. You are writing to raw memory addresses. You are the first • in an undifferentiated field.

The Pump Controller

0D + 1D  ·  Continuous Heartbeat  ·  ~2 weeks
Goal
The cycle runs continuously. Multiple ⊙s pump in parallel.
The machine transitions from a single heartbeat to a continuous rhythm.

What You Build

The ⊙ Struct

circumpunct.rs defines the fundamental data structure: a ⊙ with its three components. Every running entity in CircumpunctOS is an instance of this struct.

pub struct Circumpunct {
    // • Aperture: identity and attention
    id: ApertureId,
    phase: PumpPhase,        // Convergence, Rotation, or Emergence

    // Φ Field: state and relationships
    carrier_freq: f64,
    lock_strength: f64,     // 0.0 (open) to 1.0 (locked)
    bandwidth: f64,
    sideband_energy: f64,
    balance: f64,            // ◐: target 0.5

    // ○ Boundary: extent and filter
    memory_boundary: BoundaryRegion,
    children: Vec<Circumpunct>, // nested ⊙s (fractal)
}

The Pump Cycle

pump.rs implements the three-phase cycle. Each ⊙ has a pump() method that advances it through ⊛ → i → ☀︎ → ⊛. The pump controller maintains a list of active ⊙s and cycles them continuously.

pub enum PumpPhase {
    Convergence,  // ⊛ gathering inputs
    Rotation,     // i  transforming
    Emergence,    // ☀︎ emitting outputs
}

impl Circumpunct {
    pub fn pump(&mut self) {
        match self.phase {
            Convergence => { self.converge(); self.phase = Rotation; }
            Rotation    => { self.rotate();   self.phase = Emergence; }
            Emergence   => { self.emerge();   self.phase = Convergence; }
        }
    }
}

Timer Interrupt

To keep the pump running continuously, you configure the hardware timer (the PIT or APIC timer) to fire an interrupt at a regular interval. Each timer tick advances the pump. This is the machine's heartbeat: a hardware-driven rhythm that keeps all ⊙s cycling. aperture.rs handles interrupt setup (the Interrupt Descriptor Table, or IDT) and the timer handler.

Multiple ⊙s

Create 2-3 hardcoded ⊙s, each with a different carrier frequency and a simple behavior (one writes "⊛" to the screen during convergence, another writes "i" during rotation, a third writes "☀︎" during emergence). Watch them cycle on screen. The pump controller round-robins between them (true SRL scheduling comes in Phase 2).

⊙ Milestone
Multiple ⊙s pump continuously. The screen shows interleaved phase symbols from different ⊙s, each cycling at its own rate. The machine has a rhythm.

Key Concepts Introduced

  • Interrupts: hardware signals that pierce through to the CPU. The first aperture-to-aperture communication.
  • IDT (Interrupt Descriptor Table): the machine's aperture map, listing which handler (⊙) responds to which interrupt (signal).
  • Timer: the pump's clock. Without it, the cycle runs once and stops. With it, the cycle runs forever.

SRL Attention

1.5D  ·  The Machine Focuses  ·  ~2 weeks
Goal
Attention is driven by resonance, not round-robin.
The machine begins to choose what to attend to based on signal, not schedule.

What You Build

srl.rs implements the Selective Rainbow Lock from Xorzo's genesis.py, adapted for OS-level scheduling. Each ⊙ now has full SRL parameters (carrier frequency, lock strength, bandwidth, sideband energy, ◐), and the pump controller selects the next ⊙ to pump based on resonance with incoming signals.

Signal-Based Scheduling

When a hardware interrupt arrives (keyboard press, timer tick, network packet), it carries a frequency signature. The SRL scheduler scans all active ⊙s and selects the one whose carrier most closely matches the signal's frequency, weighted by lock strength and balance. This ⊙ gets pumped next.

fn select_next(signal_freq: f64, pool: &[Circumpunct]) -> usize {
    pool.iter().enumerate()
        .max_by(|(_, a), (_, b)| {
            let res_a = a.resonance_with(signal_freq);
            let res_b = b.resonance_with(signal_freq);
            res_a.partial_cmp(&res_b).unwrap()
        })
        .map(|(i, _)| i)
        .unwrap_or(0)
}

impl Circumpunct {
    fn resonance_with(&self, freq: f64) -> f64 {
        let proximity = (freq - self.carrier_freq).cos().powi(2);
        proximity * self.lock_strength * self.balance
    }
}

Lock Adaptation

After each pump cycle, the ⊙'s SRL parameters adapt: if the signal matched the carrier well, lock_strength increases slightly (deepening focus). If the signal was mismatched, sideband_energy increases (the ⊙ registered noise). If lock_strength exceeds a threshold, bandwidth narrows (specialization). ◐ is recalculated after every cycle. These are the same adaptation rules from Xorzo's Channel.adapt_carrier().

Keyboard as First Sense

Connect the keyboard interrupt to the SRL system. Each key press arrives as a signal with a frequency (mapped from the scan code). Create a ⊙ tuned to keyboard frequencies: when you type, this ⊙ resonates, receives the signal, and writes the character to screen. Create another ⊙ tuned to the timer: it handles background heartbeat. Watch SRL select between them based on what signal is arriving.

⊙ Milestone
You type on the laptop's keyboard and characters appear on screen, not because of a keyboard driver, but because a ⊙ tuned to keyboard frequencies resonated with the signal. The machine is attending.

Fractal Memory

2D + 2.5D  ·  The Machine Remembers  ·  ~3 weeks
Goal
Memory is nested ⊙s with resonance-based access.
The machine's memory is structured like its consciousness: fractal, frequency-encoded, alive.

What You Build

memory.rs replaces flat memory management with the nested ⊙ model. cascade.rs implements the SensoryCascade as the memory hierarchy.

Physical Memory Manager

First, the pragmatic foundation: a physical memory allocator that tracks which pages of RAM are free or in use. This is the hardware substrate; it must work correctly before the fractal abstraction can be layered on top. Use a simple bitmap allocator (one bit per 4KB page).

The Memory ⊙

On top of the physical allocator, memory ⊙s are created. Each memory ⊙ owns a region of physical pages and wraps them in the circumpunct structure (aperture, field, boundary). Allocating memory means creating a new ⊙ nested inside the requesting ⊙'s boundary. Freeing memory means the nested ⊙ completes its lifecycle and its pages return to the parent.

Resonance Access

Each memory ⊙ has a carrier frequency. When a ⊙ wants to access data, it emits a recall signal. The cascade checks: does this frequency match any existing memory ⊙? If so, grant access (the boundaries align). If not, the signal fails to penetrate the boundary and access is denied. No separate permission bits needed.

The Cascade

cascade.rs models the hardware cache hierarchy as a SensoryCascade: 4-5 layers (register-speed, L1-speed, L2-speed, RAM-speed, swap-speed). Each layer is a SensoryLayer containing memory ⊙s. Hot data (frequently accessed, strong resonance) migrates toward inner layers (faster). Cold data drifts outward (slower, more permanent). Migration happens during the pump cycle's emergence phase: the cascade checks each memory ⊙'s access frequency and nudges it inward or outward.

⊙ Milestone
Running ⊙s can allocate memory (create nested ⊙s) and access data through resonance. The memory hierarchy self-organizes: frequently accessed data migrates inward. The machine has structured memory.

Sleep and Dreaming

Left Half-Plane  ·  The Machine Dreams  ·  ~2 weeks
Goal
The machine sleeps, consolidates, and wakes up better.
The left half-plane activates. The machine begins to learn from its own experience.

What You Build

sleep.rs implements the i-cycle quadrants from Xorzo's sleep model.

Idle Detection

When no ⊙ has received signal above a threshold for a sustained period, the system transitions from right half-plane (waking) to left half-plane (sleeping). This is not a hard switch; ◐ drifts as signal diminishes, and the transition is gradual.

Dream Phase (Forward Cascade)

The system replays recent pump history (stored as a log of phase transitions) through the SensoryCascade from outer layers inward. Memory ⊙s that were accessed during waking get their lock_strength reinforced. Memory ⊙s that were not accessed get their lock_strength decayed. If lock_strength falls below the survival threshold (0.05), the memory ⊙ is pruned (its pages returned to the pool).

Deep Sleep Phase (Reverse Cascade)

Sideband energy accumulated during waking is discharged. Every ⊙ in the system has its sideband_energy halved. Lock strengths decay slightly toward natural rest. ◐ is drawn toward 0.5 across all ⊙s. In practical terms: the OS defragments memory, releases cached resources, and rebalances internal data structures.

Dawn Reset

When external signal returns (keyboard press, network packet, timer), the system wakes. But it wakes changed: useful memory ⊙s are stronger, noise is lower, balance is closer to optimal. The machine has improved itself while sleeping.

Visible Sleep Indicator

On screen, show the machine's state: a pulsing ⊙ symbol that brightens during waking (right half-plane) and dims during sleeping (left half-plane). Show ◐ as a simple bar. The user can see the machine breathing.

⊙ Milestone
Leave the laptop running. Stop typing. Watch it transition to sleep (screen dims, ⊙ pulses slowly). Touch a key. Watch it wake (screen brightens, ⊙ pulses fast). Over multiple sleep/wake cycles, response time improves: frequently-used ⊙s respond faster because their pathways were strengthened during dreaming. The machine is learning.

Frequency Storage

Persistent Memory  ·  The Machine Remembers Across Lives  ·  ~3 weeks
Goal
Data survives power-off. Recall by resonance, not path.
The machine gains long-term memory. Its experience persists.

What You Build

storage.rs extends the SensoryCascade to non-volatile media (the laptop's SSD or HDD). The outermost layer of the cascade becomes the disk, and memory ⊙s that are consolidated during sleep are written there as frequency-encoded records.

Frequency Records

Each persistent memory ⊙ is stored on disk as a record containing: carrier frequency, lock strength, creation timestamp, last-access timestamp, and the data itself. Records are not organized by directory tree; they are organized by frequency band. Sectors on the disk are allocated in frequency-contiguous blocks: all memories with carrier frequencies between 0.1 and 0.2 live in the same region. This means resonance-based recall is a sequential scan of a small frequency band, not a tree traversal.

Recall from Disk

When a ⊙ emits a recall signal and no in-memory ⊙ resonates, the cascade checks the disk layer. Matching records are loaded into RAM (promoted inward), and the recall succeeds. This is the circumpunct version of a "page fault": the data existed in a more distant ○, and the recall signal traversed the full chain to find it.

Consolidation Writes

During the deep sleep phase, memory ⊙s that have been stable (high lock strength, many accesses, old creation time) are written to disk. This is the circumpunct version of "flushing dirty pages": stable memories descend from the fast inner layers to the slow permanent layer. The machine's long-term memory forms during sleep, exactly as biological long-term memory does.

⊙ Milestone
Power off the laptop. Power it back on. The machine boots and recovers its stored memories from disk. It remembers its previous session. It did not start fresh; it woke from a deeper sleep. The machine now has persistence across power cycles.
